0. Bootstrap The Workflow Lane

Create the local workflow lane, establish canonical state, seed team/runtime files, and prepare transaction-safe command handling.

Primary commands

wrkflw:discuss, companion init script, command handler prepare phase.

Artifacts

state.md, history.md, gates.md, workflow-contract.md, diagrams, team files.

Decision

Establish the lane without silently skipping gates or OpenSpec requirements.

Inner steps
  • Create `.workflow/<slug>/` and register the lane in `.workflow/initiative-index.md`.
  • Initialize `state.md` with canonical stage and gate values, not freeform status labels.
  • Seed coordination artifacts such as `agent-assignments.md`, `execution-board.md`, `runtime-contract.md`, `team-minutes.md`, and `review-log.md`.
  • Create or refresh `diagram-flow.puml` and `diagram-work.puml` after state-changing commands.
  • Run workflow commands through a transaction journal so failed commands can roll back and `wrkflw:resume` can continue safely.
Command surface
wrkflw:discuss wrkflw:actions wrkflw:resume wrkflw:next wrkflw:override wrkflw:accounting-record
repo-first discovery design normalization

1. Design Document And Codebase Analysis

Inspect the existing repository before treating a design seed as the source of truth. Normalize broad designs into workflow-ready slices and surface drift early.

Inputs

README.md, build files, source entrypoints, tests, active OpenSpec, design.md or docs/design.md.

Artifacts

design-seed.md, design-slice.md, .workflow/_normalized/master-design.md, epic-candidates.md, links.md.

Checks

Code vs design drift, code vs workflow drift, code vs OpenSpec drift, current implementation shape, coverage gaps.

Inner steps
  • Read existing repo evidence first, especially README, package/build files, source roots, tests, and active OpenSpec changes.
  • Auto-detect a design seed from `design.md` or `docs/design.md`, or use an explicitly supplied design file.
  • If the design is broad, normalize it into shared design material and an epic-specific `design-slice.md`.
  • Preserve section semantics such as actors, capabilities, architecture, operations, and constraints so story shaping does not flatten everything into one backlog.
  • If the implementation is ahead of workflow artifacts, classify it as reconciliation work before new planning continues.
Commands and recommended routing
wrkflw:discuss "..." wrkflw:reconcile "..." wrkflw:openspec-sync wrkflw:refine "..."
  • Use `reconcile` when repo evidence is ahead of `.workflow` or OpenSpec artifacts.
  • Use `openspec-sync` when the active workflow is valid but the OpenSpec change is stale.
  • Use `refine` when the analysis is directionally right but missing an important artifact or concern.
human gate capability inventory

2. Capability Review

Review the generated capability inventory before epic shaping. This is the first guard against thin planning or accidental epic-wide scope.

Primary artifact

capabilities.md with workflow mode, required/recommended/optional capabilities, and story prompts.

Human decision

Approve, reject, refine, rework, or defer explicit capability scope.

Planning effect

Approved capabilities seed `stories.md`; they also flow into OpenSpec coverage context.

Capability modes
  • `tutorial-sample`: optimize for pedagogy and a clear learning path.
  • `feature-harness`: optimize for broad coverage and realistic feature comparison.
  • `product-service`: optimize for runtime/service behavior and realistic execution paths.
  • `general-delivery`: default when no stronger mode signal exists.
Commands
wrkflw:approve "..." wrkflw:reject "..." wrkflw:refine "..." wrkflw:rework "..." wrkflw:defer "..."
human gate problem framing

3. Epic Shaping

Turn analysis and capabilities into an epic boundary with explicit goals, non-goals, assumptions, risks, constraints, and decisions.

Artifacts

context.md, decisions.md, assumptions.md, risks.md, links.md.

Human decision

Confirm the epic boundary before slicing stories.

Risk

Unclear non-goals here become scope drift during OpenSpec or implementation planning.

Review focus
  • Does the epic match the codebase reality discovered earlier?
  • Are non-goals explicit enough to prevent overbuilding?
  • Are assumptions visible instead of hidden inside story text?
  • Are dependencies on other lanes captured in `dependencies.md`?
human gate SWE-AF: issue DAG

4. Story Slicing And Executable DAG

Split the epic into small, mergeable stories and turn story dependencies into a schedulable DAG with ready, blocked, deferred, active, and completed status hints.

Artifacts

stories.md, dependencies.md, dag.json, dag.md, dag-validation.md.

SWE-AF influence

Executable story DAG, dependency levels, ready nodes, parallelism eligibility.

Human decision

Approve the slice set or rework stories that are too broad, too thin, or dependency-unsafe.

Inner steps
  • Generate `stories.md` from the reviewed capability inventory.
  • Extract `Depends on:` lines and lane blockers into the DAG.
  • Validate unknown dependencies, dependency cycles, and lane-level blockers.
  • Compute execution levels so safe parallel dispatch can be prepared later.
  • Carry debt warnings and high-risk story metadata into planning context.
Commands
wrkflw:approve "..." wrkflw:dag-sync wrkflw:proceed-only "Story N" wrkflw:defer "Story N" wrkflw:rework-item "Story N"
human gate per story planner risk metadata

5. Story Enrichment

For the active story, add acceptance criteria, test expectations, allowed write paths, risk metadata, and review focus before spec work starts.

Artifact

story-N.md with scope, ACs, tests, risks, likely changed paths, and ownership notes.

Inherited context

DAG dependencies, downstream impact, memory, debt, prior findings, and design slice.

Parallel safety

Ready stories need explicit disjoint Allowed Write Paths for parallel dispatch.

SWE-AF-style risk metadata
  • Estimated scope and likely changed paths.
  • Touches interfaces, database, auth, transport, side effects, migrations, or cross-module contracts.
  • Needs new tests or deeper QA.
  • Risk rationale and review focus for role packets.
Supporting commands
wrkflw:memory-record wrkflw:debt-record wrkflw:refine wrkflw:approve
human gate OpenSpec lane

6. OpenSpec Authoring And Review

Create or update a story-scoped OpenSpec change and block if OpenSpec is required but unavailable, unless the user explicitly waives it.

Artifacts

proposal.md, spec.md, tasks.md under openspec/changes/<change>/.

Gate

Human reviews the proposal/spec/tasks before implementation planning.

Guardrail

OpenSpec is single-lane by default and should not pre-create every epic change.

Coverage controls
  • Proposal carries workflow mode and capability categories intentionally covered by the active story.
  • Spec carries explicit current-story scenarios and deferred coverage visibility.
  • Tasks make capability coverage explicit and leave follow-up context for deferred categories.
Commands
wrkflw:openspec-sync wrkflw:override "..." wrkflw:approve "..." wrkflw:reject "..."
human gate risk-based execution path

7. Implementation Planning

Turn the approved story/spec into an executable PR-sized plan with team ownership, DAG context, risk route, validation expectations, and explicit file boundaries.

Artifacts

implementation-plan.md, execution-path.json, execution-path.md, assignments.

Routing

`simple` for implementer/reviewer; `flagged` for Tech Lead, Reviewer QA, and fresh synthesis.

Gate concern

The first slice must include required acceptance criteria, not defer them unless the story explicitly says so.

Plan completeness checks
  • Every acceptance criterion needs implementation and validation coverage.
  • Test-only slices are invalid when the runtime skeleton does not exist.
  • High-risk/interface stories should not defer object rules, classifier outputs, or contract behavior that are acceptance criteria.
  • Team ownership must respect disjoint write scopes when parallel execution is planned.
Commands
wrkflw:execution-path wrkflw:staff wrkflw:assign wrkflw:approve wrkflw:refine
SWE-AF-inspired runtime delegated team execution

8. Team Runtime, Dispatch, And Parallel Worktrees

Prepare role packets and optional isolated worktrees so Codex can run Product Owner, Tech Lead, Implementer, and Reviewer QA lanes with explicit sync back into workflow artifacts.

Artifacts

team-dispatch.md, dispatch/*.md, parallel-dispatch/*, worktrees/manifest.json.

Result ingestion

agent-results/*, agent-sync-ledger.md, strict result schema validation.

Team model

Product Owner, Tech Lead, Implementer lanes, Reviewer QA, with role reviews and conflict records.

Delegated execution sequence
  • `team-run` generates role packets for the active story and can prepare merge-eligible implementer worktrees.
  • `team-run-level` dispatches every safe ready story in the earliest DAG level when at least two ready stories have disjoint allowed paths.
  • Agents return structured result envelopes into `agent-results/`.
  • `team-sync-all` validates and ingests envelopes, updates `role-reviews.md`, `review-log.md`, `conflicts.md`, `assumptions.md`, `team-minutes.md`, and execution board state.
  • `team-sync` can still ingest direct lightweight role progress when a full envelope is unnecessary.
Commands
wrkflw:team-run wrkflw:team-run-level wrkflw:team-sync wrkflw:team-sync-all wrkflw:challenge wrkflw:review-sync wrkflw:worktree-clean
merge and validation gates evidence before review

9. Merge Gate, Merge Apply, Integration Gate, CI, And Verify-Fix

After implementation, wrkflw checks changed paths, applies approved lane branches only with confirmation, records integration validation, captures external CI, and verifies acceptance criteria.

Artifacts

merge-gate.md, merge-apply.md, integration-test-gate.md, ci-feedback.md, verify-fix.md.

Safety

Merge gate is read-only; merge apply requires confirm: merge-apply; integration execution is allowlisted.

Blocking

Review approval blocks on stale, missing, failed, or waived-without-reason validation evidence.

Validation layers
  • `merge-gate` verifies worktree cleanliness, changed paths, conflict probes, manifest freshness, and ownership scope.
  • `merge-apply` applies only ready wrkflw-owned lane branches after explicit confirmation and records a pre-apply checkpoint.
  • `integration-gate` decides whether integration validation is required and can run only allowlisted `test-id` entries with structured argv.
  • `ci-feedback` records external CI status against the current `HEAD` and classifies failures.
  • `verify-fix` compares story acceptance criteria against explicit evidence and creates focused fix tasks for failed or unverified criteria.
Commands
wrkflw:merge-gate wrkflw:merge-apply "confirm: merge-apply" wrkflw:integration-gate wrkflw:integration-gate "test-id: api-smoke" wrkflw:ci-feedback wrkflw:verify-fix
human gate feedback synthesizer

10. Review Gate And Feedback Synthesis

Synthesize role reviews, findings, conflicts, debt, execution path, validation gates, CI, and failure classifications into a single actionable recommendation.

Artifacts

feedback-synthesis.md, role-reviews.md, review-log.md, conflicts.md, assumptions.md.

Recommendations

approve, fix, split, defer, block, or replan.

Gate behavior

Flagged execution paths require fresh approving synthesis before release planning.

Review evidence
  • Reviewer QA and Product Owner evidence can be required by gate configuration.
  • Open blocking conflicts prevent gate advancement until resolved.
  • Incorrect assumptions and contested assumptions are kept visible in `assumptions.md`.
  • Promoted failure classifications can route the story to fix, split, block, or replan.
Commands
wrkflw:feedback-synth wrkflw:review-sync wrkflw:challenge wrkflw:approve wrkflw:reject
recovery loop SWE-AF: issue advisor and replanner

11. Recovery, Issue Advisor, Replan, Debt, And Memory

When the story is blocked, wrkflw records typed failure evidence, recommends a bounded recovery action, and can propose human-approved DAG/story mutations.

Artifacts

issue-advisor.md, replan.md, debt.md, memory.md, append-only records.

Advisor actions

retry_approach, retry_modified, accept_with_debt, split, escalate_to_replan.

Mutation rule

Replan applies only with confirm: replan and preserves completed story history.

Recovery routing
  • Failure classifications identify dependency blocks, broad scope, environment failures, design contradictions, test failures, policy/security blocks, and merge/integration problems.
  • `issue-advisor` consumes DAG, feedback synthesis, review evidence, debt, memory, gate artifacts, verify-fix, CI, and promoted failure classes.
  • `replan` can propose split stories, modified acceptance criteria, skip/defer, remove, dependency rewrite, and remaining-story reorder.
  • `debt-record` keeps accepted gaps visible and blocks release when unresolved high/critical debt remains open.
  • `memory-record` captures repo conventions, failure patterns, interface notes, validated test commands, and implementation patterns for later stories.
Commands
wrkflw:issue-advisor wrkflw:replan wrkflw:replan "confirm: replan" wrkflw:debt-record wrkflw:memory-record wrkflw:resume
human gate story closeout

12. Release Planning And Story Closeout

Review production readiness, remaining debt, release notes, validation status, and signoff evidence before marking the story complete.

Artifacts

release-plan.md, history.md, archived OpenSpec change, updated diagrams.

Blocks

Open high/critical debt, stale verification, missing required signoff, or stale OpenSpec sync.

Outcome

Story reaches `done`; completed progression remains visible in diagrams and history.

Closeout checks
  • Confirm whether the story is production-worthy or local-only progress.
  • Resolve, accept, or explicitly carry remaining debt.
  • Archive the OpenSpec change only after the story closes.
  • Clean wrkflw-owned worktrees when safe.
Commands
wrkflw:approve wrkflw:debt-record wrkflw:openspec-sync wrkflw:worktree-clean
loop or exit DAG scheduler

13. Next Story Loop Or Workflow Complete

After each story closes, wrkflw checks the DAG for ready remaining stories. If one exists, it loops back to story enrichment; if none remain, the workflow exits.

Loop: Story Done -> DAG status check -> `wrkflw:proceed-only "Story N"` or first ready story -> Story Enrichment -> OpenSpec -> Implementation -> Review -> Release.
Exit: no ready or remaining story for the lane -> workflow state becomes `done`.
What repeats per story
  • Story enrichment and approval.
  • OpenSpec story-scoped change and approval.
  • Implementation planning and approval.
  • Implementation/team dispatch/validation/review/recovery as needed.
  • Release planning and story closeout.
Cross-cutting records updated throughout
  • `state.md`, `history.md`, `execution-board.md`, `diagram-flow.puml`, `diagram-work.puml`.
  • `records/*.jsonl`, `accounting.md`, `memory.md`, `debt.md`, and transaction checkpoints.
  • OpenSpec lane ownership and archive state.

This map intentionally treats wrkflw as an artifact-first, human-gated workflow control plane with selected SWE-AF-inspired mechanics. It does not claim that Python scripts autonomously spawn agents; they prepare evidence, packets, records, and gates that Codex or another runtime can use.
